## 场景 1：找谁读写 + 生成 AOB 注入脚本

> 用户：0x1A2B3C4D 是金币地址，我想找出谁在写这个地址，然后生成注入脚本实现修改

**建议方案：** CE 中右键该地址 → "找出是什么改写了这个地址" → 触发后你会看到写入指令，格式如 `game.exe+0x3A2B10 - mov [rcx+08], eax`。

拿到指令后有两条路：

**A. 简单注入 — 直接在指令处 hook：**
> 对 `game.exe+0x3A2B10` 做代码注入，修改 eax 为 9999，然后用该地址生成 AOB 脚本。CE 菜单 → Tools → Auto Assemble → AOB Injection 可以自动生成模板。

**B. 注意区分路径：**
> 如果你看到同一条指令被多次触发，结合"找出是什么访问了这个地址"看调用栈，区分不同触发场景。比如同一条 `mov [rcx+08], eax` 可能被 Shop::Buy（扣钱）和 Quest::Reward（加钱）同时调用。你需要确定要 hook 哪个路径。

**生成的脚本模板：**
```
[ENABLE]
aobscanregion(GoldHook, game.exe+0x3A2B00, game.exe+0x3A2B20, 89 41 08 xx xx xx)
label(code)
label(return)
registersymbol(GoldHook)

GoldHook:
  mov [rcx+08], #9999    -- 把金币固定为 9999
  jmp return

code:
  mov [rcx+08], eax       -- 原始指令
  jmp return

GoldHook+07:
  db 90 90 90

[DISABLE]
GoldHook:
  db 89 41 08 xx xx xx
unregistersymbol(GoldHook)
```

## 场景 2：同一条指令访问了多个地址

> 用户：game.exe+0x3A2B10 - mov [rcx+08], eax 这条指令访问了 342 个地址。0x1A2B3C4D 是金币，我该怎么单独改金币不影响其他值？

**分析：** 这是 C++ 游戏里常见的模式——一个通用的 SetField 函数被所有属性复用。金币、血量、经验值可能都走同一行代码，区别在于 `rcx`（this 指针）指向不同对象，或 `rcx+08` 偏移量在不同对象上映射不同字段。

**建议方案：**

1. 先确认金币是哪个对象：看 `rcx` 的值，然后在 CE 中对该值再次搜索，看有哪些指针指向它
2. 在注入脚本中加判断：对比 `rcx` 是否为 Player 对象的基址，只修改 Player 的金币

```
GoldHook:
  cmp rcx, [PlayerBase]     -- 比较是不是 Player 对象
  jne original               -- 不是就走原始逻辑
  mov [rcx+08], #9999        -- 是 Player，金币改 9999
  jmp return
original:
  mov [rcx+08], eax
  jmp return
```

3. 如果区分不了对象，另一种方案是找**调用者**区分——在 hook 中读返回地址，根据是谁调用的来决定是否修改

## 场景 3：分析汇编找基址

> 用户：0x1A2B3C4D 是金币，但每次重启地址都变，帮我找基址。不想用指针扫描。

**分析方案：**

**第 1 步：找到写入指令，分析地址计算公式**

对 0x1A2B3C4D 设"找出是什么改写了这个地址"。触发后看到：
```
game.exe+0x3A2B10 - mov [rcx+08], eax
```
寄存器: `rcx = 0x1A2B3C45`, `eax = 9999`
→ 金币地址 = `rcx + 0x08`，当前 `rcx = 0x1A2B3C45`

**第 2 步：向上追 rcx 的来源**

在 `game.exe+0x3A2B10` 之前几行找 `rcx` 被赋值的地方：
```
game.exe+0x3A2B08 - mov rcx, [rdx+20]
```
→ rcx 来自 `[rdx + 0x20]`，接下来需要找到 rdx 的值

**第 3 步：追 rdx，可能跨函数**

rdx 在当前函数内没有被修改 → 需要往上到调用者中找。

记忆断点当前触发时的 rdx 值，然后在调用者（通过调用栈找到，如 `game.exe+0x5A0000`）的函数体里面，从下往上找 `mov rdx, ...` 或者 `lea rdx, [...]` 的指令。

**第 4 步：找到静态地址即停止**

如果在调用者中找到：
```
game.exe+0x5A0060 - mov rdx, [game.exe+0x7F0000]
```
→ `game.exe+0x7F0000` 是静态地址，追溯结束。

**基址链：**
```
[[game.exe+0x7F0000] + 0x20] + 0x08
```

**如果当前函数和调用者都没找到怎么办：**

继续往调用者的调用者翻。如果翻了几层都是 `[reg+offset]` 但的寄存器始终来自上一层的参数，一路追到最外层必然有个 `mov reg, [static_address]`。这个过程 CE 的"指针扫描"就是自动化做的，手动一样能做，只是费时间。

## 场景 4：加密数值搜不到

> 用户：游戏显示金币 100，CE 搜 100 搜不到任何结果

**分析：** 数值在内存中被加密存储了。加密后无法用精确值或增减量来过滤——显示从 100 变 90，但加密值的变化量完全不可知。

**建议方案 — 变/不变过滤链：**

```
第 1 轮: CE 扫描类型选"未知初始值"，4 字节
          → 会有几十上百万个结果

第 2 轮: 花钱（确保金币变了），扫描类型选"Changed value"
          → 排除所有没变的地址

第 3 轮: 逛街或挂机（金币不变），选"Unchanged value"
          → 进一步排除

第 4 轮: 再花钱，再选"Changed value"
          → 重复几轮，候选会快速缩小到几个地址
```

这个过程依赖的核心逻辑是：金币变你就选 changed，金币不变你就选 unchanged。不需要知道具体数值。

**确认地址后反推加密算法：**

1. 拿到候选地址（如 0x1B2C3D4E），记下当前显示值和存储值
2. 手动修改存储值，观察游戏显示怎么变：
   - 改成 0 → 显示 0？说明没有 XOR（XOR 0 ≠ 0）
   - 改大一倍 → 显示大一倍？说明是简单乘法
   - 改成 0 → 显示不是 0？说明含 XOR 或偏移
3. 最多试 3-4 个值，规律就出来了

**找到加密函数（如果不想手动试）：**

对加密值地址设写入断点，找到写入指令后往上翻几行看加密逻辑：
```
game.exe+0x3A2B10 - mov [rcx+08], eax   ← 写入加密值
game.exe+0x3A2B08 - xor eax, 0x12345678  ← 加密操作
game.exe+0x3A2B04 - mov eax, [ebp-0x04]  ← 真实值
```

之后注入脚本只需要在写入前先加密：`要写的值 = 9999 xor 0x12345678`。
